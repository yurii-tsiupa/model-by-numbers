# Internationalization

The application uses a lightweight client-side provider because locale-prefixed routes and server translation middleware are unnecessary for the current App Router architecture. `I18nProvider` is mounted once in the root layout, so changing language does not remount editor state or write project data.

English (`en.ts`) defines `TranslationKey`; Ukrainian (`uk.ts`) must satisfy the same dictionary shape. Add a key to English, add its Ukrainian value, then call `t("the.key", { value })`. Interpolation uses `{value}` placeholders.

Use `formatCount` for counts. It delegates to `Intl.PluralRules`, including Ukrainian forms such as `1 –¥–µ—Ç–∞–ª—å`, `2 –¥–µ—Ç–∞–ª—ñ`, and `5 –¥–µ—Ç–∞–ª–µ–π`. Use `formatLocalizedDate` and `formatLocalizedNumber` instead of implicit `toLocale*` calls.

To add a locale, extend `Locale`, constants, the dictionary map, Intl locale map, and count forms. Then add it to `LanguageSwitcher`.

New guides receive the active locale in `buildGuideData`. Preview and PDF components read `guide.locale`; old snapshots without it default safely to English. This makes historical PDFs deterministic after the application language changes. The registered Roboto files support Cyrillic text.

Never persist translated domain labels or use the ambient UI locale to render a historical snapshot.
